# Лабораторная работа №4 – Интерфейсы и абстрактные классы (ABC)

## 1. Цель работы

Изучение механизма абстрактных базовых классов (ABC) в Python, освоение понятия интерфейса как контракта поведения. Практическое применение интерфейсов для обеспечения полиморфизма без привязки к конкретным классам, а также для создания расширяемой архитектуры.

## 2. Описание интерфейсов

В рамках работы созданы два абстрактных базовых класса (интерфейса) в файле `interfaces.py`:

| Интерфейс | Абстрактный метод | Назначение |
|-----------|-------------------|------------|
| `Printable` | `to_string() -> str` | Возвращает строковое представление объекта (специализированное, не зависящее от `__str__`). |
| `Comparable` | `compare_to(other) -> int` | Сравнивает текущий объект с другим. Возвращает отрицательное число, 0 или положительное число в зависимости от того, меньше, равен или больше текущий объект. |

Оба интерфейса наследуются от `ABC` и используют декоратор `@abstractmethod`.

## 3. Реализация в классах

Интерфейсы реализованы в следующих классах (файл `models.py`), которые наследуются от существующих классов из ЛР‑3:

| Класс | Наследует от | Реализуемые интерфейсы | Поведение методов |
|-------|--------------|------------------------|-------------------|
| `PrintableAthlete` | `Athlete` (из ЛР‑3) | `Printable` | `to_string()` возвращает строку с именем, весом и рекордом. |
| `ComparableCompetitiveAthlete` | `CompetitiveAthlete` (из ЛР‑3) | `Comparable` | `compare_to()` сравнивает двух соревновательных спортсменов по их рейтингу (`rating`). |
| `MultiInterfaceAthlete` | `RecreationalAthlete` (из ЛР‑3) | `Printable`, `Comparable` | `to_string()` выводит имя, любимое занятие и уровень удовольствия; `compare_to()` сравнивает по уровню удовольствия (`enjoyment_level`). |

Таким образом, разные классы реализуют одни и те же интерфейсы по‑разному, что демонстрирует полиморфизм.

## 4. Демонстрация

Демонстрационный скрипт `demo.py` содержит 4 сценария, показывающих работу интерфейсов. Ниже приведены скриншоты выполнения и описание каждого сценария.

### Сценарий 1. Реализация интерфейсов в разных классах

Создаются объекты `PrintableAthlete`, `ComparableCompetitiveAthlete` и `MultiInterfaceAthlete`. Вызываются методы `to_string()` и `compare_to()`. Результат показывает, что каждый класс даёт своё строковое представление и свою логику сравнения.

### Сценарий 2. Интерфейс как тип, универсальная функция, множественная реализация

- Демонстрируется проверка `isinstance(obj, Printable)`.  
- Написана универсальная функция `print_all(items)`, которая принимает список любых объектов, но для `Printable` вызывает `to_string()`, а для остальных выводит стандартное представление.  
- Показан класс `MultiInterfaceAthlete`, реализующий оба интерфейса.

![Сценарий 2](/src/images/lab04_img/img_lab04_scen2.png)

### Сценарий 3. Коллекция `AthleteCollection` и фильтрация по интерфейсу

В коллекцию (из ЛР‑2) добавляются объекты разных типов. Затем коллекция фильтруется по наличию интерфейсов `Printable` и `Comparable` с помощью `isinstance`. Для отфильтрованных объектов вызываются методы интерфейсов.

![Сценарий 3](/src/images/lab04_img/img_lab04_scen3.png)

### Сценарий 4. Сортировка списка объектов через интерфейс `Comparable`

Создаётся список `ComparableCompetitiveAthlete`. Реализована сортировка пузырьком, которая использует метод `compare_to()` для сравнения элементов. Список сортируется по возрастанию рейтинга.

В лабораторной работе №4 успешно созданы и применены абстрактные интерфейсы `Printable` и `Comparable`. Реализованы классы, наследующие существующую иерархию спортсменов и реализующие эти интерфейсы. Продемонстрированы: использование интерфейса как типа, универсальные функции, множественная реализация интерфейсов, фильтрация коллекции по интерфейсу и сортировка через `Comparable`.
